### Pod Status

Pods can be in one of several phases:

<AccordionGroup>
  <Accordion icon="circle-check" title="Running">
    Pod is running normally. All containers started successfully.

    **Indicator:** Green badge with "Running"
    
    **What it means:** The pod has been bound to a node and all containers have been created. At least one container is running or is in the process of starting or restarting.
  </Accordion>

  <Accordion icon="clock" title="Pending">
    Pod is waiting to be scheduled or containers are being created.

    **Indicator:** Yellow badge with "Pending"
    
    **Common reasons:**
    - Waiting for node assignment
    - Pulling container images
    - Insufficient cluster resources
    - Volume mounting in progress
  </Accordion>

  <Accordion icon="xmark-circle" title="Failed">
    Pod has failed and will not restart.

    **Indicator:** Red badge with "Failed"
    
    **Common reasons:**
    - Container exited with error code
    - Image pull failed permanently
    - Node failure
    - Resource constraints
  </Accordion>

  <Accordion icon="check-circle" title="Succeeded">
    Pod completed successfully (typically from Jobs).

    **Indicator:** Blue badge with "Succeeded"
    
    **What it means:** All containers in the pod have terminated successfully and will not be restarted.
  </Accordion>

  <Accordion icon="rotate" title="CrashLoopBackOff">
    Container is repeatedly crashing.

    **Indicator:** Red badge with "CrashLoopBackOff"
    
    **Common reasons:**
    - Application startup failure
    - Missing dependencies
    - Configuration errors
    - Resource limits too low
  </Accordion>

  <Accordion icon="triangle-exclamation" title="ImagePullBackOff">
    Container image cannot be pulled.

    **Indicator:** Orange badge with "ImagePullBackOff"
    
    **Common reasons:**
    - Image doesn't exist
    - Registry authentication failed
    - Network connectivity issues
    - Wrong image name/tag
  </Accordion>

  <Accordion icon="ban" title="Evicted">
    Pod was evicted from the node.

    **Indicator:** Red badge with "Evicted"
    
    **Common reasons:**
    - Node out of resources
    - Exceeded pod memory limit
    - Node maintenance
    - Priority preemption
  </Accordion>
</AccordionGroup>

### Health Checks

View liveness and readiness probe configuration and status:

<img src="/images/pod-health-checks.png" alt="Health Checks" />

#### Liveness Probe

Determines if container is alive and should be restarted:

<AccordionGroup>
  <Accordion icon="heart-pulse" title="HTTP GET">
    Performs HTTP request to specified path and port
    
    ```yaml
    livenessProbe:
      httpGet:
        path: /healthz
        port: 8080
      initialDelaySeconds: 30
      periodSeconds: 10
    ```
  </Accordion>

  <Accordion icon="terminal" title="Exec Command">
    Executes command inside container
    
    ```yaml
    livenessProbe:
      exec:
        command:
        - cat
        - /tmp/healthy
      initialDelaySeconds: 5
      periodSeconds: 5
    ```
  </Accordion>

  <Accordion icon="plug" title="TCP Socket">
    Attempts TCP connection to specified port
    
    ```yaml
    livenessProbe:
      tcpSocket:
        port: 8080
      initialDelaySeconds: 15
      periodSeconds: 20
    ```
  </Accordion>
</AccordionGroup>

#### Readiness Probe

Determines if container is ready to accept traffic:

- **Success**: Container added to service endpoints
- **Failure**: Container removed from service endpoints
- **Configuration**: Similar to liveness probe

<Warning>
  Failed readiness probes don't restart the container - they only remove it from service load balancing
</Warning>

#### Startup Probe

For slow-starting containers:

- Disables liveness/readiness checks until first success
- Prevents premature restart during startup
- Useful for legacy applications with long initialization

### Container Restart History

Track when and why containers restarted:

<img src="/images/pod-restart-history.png" alt="Restart History" />

**Information Displayed:**
- **Timestamp**: When restart occurred
- **Reason**: Why container restarted (Error, OOMKilled, etc.)
- **Exit Code**: Container exit code
- **Message**: Detailed error message
- **Signal**: Termination signal (if applicable)

**Common Exit Codes:**

| Code | Meaning |
|------|---------|
| 0 | Success (normal termination) |
| 1 | General error |
| 125 | Container failed to run |
| 126 | Command cannot be invoked |
| 127 | Command not found |
| 137 | SIGKILL (OOMKilled or killed externally) |
| 139 | SIGSEGV (segmentation fault) |
| 143 | SIGTERM (graceful shutdown) |

## Actions

### Restart Pod

Delete the pod to trigger recreation by its controller:

<Steps>
  <Step title="Click Restart Button">
    In the detail view, click **"Restart Pod"**
  </Step>

  <Step title="Confirm Action">
    A confirmation dialog appears

    <Warning>
      This deletes the pod. It will be recreated by its Deployment/StatefulSet/DaemonSet controller.
    </Warning>
  </Step>

  <Step title="Wait for Recreation">
    Watch as:
    1. Old pod enters Terminating state
    2. New pod is created (Pending)
    3. Container images are pulled
    4. New pod starts (Running)
  </Step>

  <Step title="Verify Health">
    Check new pod logs and health probes
  </Step>
</Steps>

**When to Restart:**
- Container stuck in error state
- Memory leak suspected
- Configuration changes need to be applied
- Debugging intermittent issues
- Force image re-pull

<Note>
  Standalone pods (not managed by a controller) will be permanently deleted
</Note>

## Troubleshooting

### Pod Stuck in Pending

**Symptom:** Pod remains in Pending state for extended time

**Possible Causes:**

<AccordionGroup>
  <Accordion icon="server" title="Insufficient Resources">
    Cluster doesn't have enough CPU/memory
    
    **Solution:**
    1. Check events for "Insufficient cpu" or "Insufficient memory"
    2. Scale up cluster or reduce resource requests
    3. View node capacity in Nodes page
  </Accordion>

  <Accordion icon="download" title="Image Pull Failure">
    Cannot pull container image
    
    **Solution:**
    1. Check events for image pull errors
    2. Verify image name and tag
    3. Check registry authentication
    4. Test image pull manually: `docker pull <image>`
  </Accordion>

  <Accordion icon="hard-drive" title="Volume Mount Issues">
    PersistentVolumeClaim not bound or accessible
    
    **Solution:**
    1. Check PVC status
    2. Verify StorageClass availability
    3. Check volume node affinity
  </Accordion>

  <Accordion icon="ban" title="Node Selector Mismatch">
    No nodes match pod's nodeSelector/affinity rules
    
    **Solution:**
    1. Check pod's nodeSelector in YAML
    2. Verify node labels match requirements
    3. Review affinity/anti-affinity rules
  </Accordion>
</AccordionGroup>

### CrashLoopBackOff

**Symptom:** Container repeatedly crashes and restarts

**Diagnostic Steps:**

<Steps>
  <Step title="Check Container Logs">
    View logs for error messages and stack traces
  </Step>

  <Step title="Review Exit Code">
    Check restart history for exit codes:
    - 137: OOMKilled (increase memory limit)
    - 1/127: Application error (check logs)
    - 139: Segmentation fault (application bug)
  </Step>

  <Step title="Verify Configuration">
    - Environment variables correct?
    - ConfigMaps/Secrets accessible?
    - Required files/volumes mounted?
  </Step>

  <Step title="Check Resource Limits">
    - Memory limit sufficient?
    - CPU limit not too restrictive?
    - Increase limits and test
  </Step>

  <Step title="Test Locally">
    Run container image locally with same configuration:
    ```bash
    docker run -it <image> /bin/sh
    ```
  </Step>
</Steps>

### OOMKilled (Out of Memory)

**Symptom:** Container killed with exit code 137

**Solutions:**

1. **Increase Memory Limit**
   ```yaml
   resources:
     limits:
       memory: "1Gi"  # Increase from 512Mi
     requests:
       memory: "512Mi"
   ```

2. **Analyze Memory Usage**
   - Check metrics in detail view
   - Profile application memory consumption
   - Look for memory leaks

3. **Optimize Application**
   - Reduce memory footprint
   - Implement caching strategies
   - Fix memory leaks

### ImagePullBackOff

**Symptom:** Cannot pull container image

**Common Solutions:**

<Tabs>
  <Tab title="Authentication">
    **Private Registry Authentication**
    
    Create image pull secret:
    ```bash
    kubectl create secret docker-registry regcred \
      --docker-server=<registry> \
      --docker-username=<username> \
      --docker-password=<password>
    ```
    
    Reference in pod:
    ```yaml
    imagePullSecrets:
    - name: regcred
    ```
  </Tab>

  <Tab title="Image Name">
    **Verify Image Path**
    
    Check image name format:
    - `nginx:latest` (Docker Hub)
    - `gcr.io/project/image:tag` (GCR)
    - `registry.example.com/image:tag` (Private)
    
    Common mistakes:
    - Wrong registry URL
    - Typo in image name
    - Non-existent tag
  </Tab>

  <Tab title="Network">
    **Network Connectivity**
    
    Test from node:
    ```bash
    # SSH to node
    docker pull <image>
    ```
    
    Check:
    - Internet connectivity
    - DNS resolution
    - Firewall rules
    - Proxy configuration
  </Tab>
</Tabs>

### Failed Readiness Probe

**Symptom:** Pod running but not ready (not receiving traffic)

**Solutions:**

1. **Check Probe Configuration**
   ```yaml
   readinessProbe:
     httpGet:
       path: /ready
       port: 8080
     initialDelaySeconds: 10  # Too short?
     periodSeconds: 5
     failureThreshold: 3
   ```

2. **Increase Delays**
   - Application needs more time to start
   - Increase `initialDelaySeconds`
   - Adjust `periodSeconds`

3. **Fix Health Endpoint**
   - Ensure `/ready` endpoint exists
   - Returns 200 status code
   - Responds quickly (< timeout)

4. **Check Dependencies**
   - Database connection ready?
   - External services accessible?
   - Required files loaded?

## Best Practices

<AccordionGroup>
  <Accordion icon="check" title="Always Set Resource Limits">
    Prevent resource starvation and OOMKills
    
    ```yaml
    resources:
      limits:
        memory: "512Mi"
        cpu: "500m"
      requests:
        memory: "256Mi"
        cpu: "250m"
    ```
  </Accordion>

  <Accordion icon="check" title="Configure Health Probes">
    Enable auto-recovery and traffic management
    
    ```yaml
    livenessProbe:
      httpGet:
        path: /healthz
        port: 8080
      initialDelaySeconds: 30
      periodSeconds: 10
    
    readinessProbe:
      httpGet:
        path: /ready
        port: 8080
      initialDelaySeconds: 5
      periodSeconds: 5
    ```
  </Accordion>

  <Accordion icon="check" title="Use Specific Image Tags">
    Avoid `latest` tag for reproducibility
    
    ```yaml
    # Good
    image: nginx:1.21.6
    
    # Bad
    image: nginx:latest
    ```
  </Accordion>

  <Accordion icon="check" title="Implement Graceful Shutdown">
    Handle SIGTERM for clean termination
    
    ```yaml
    lifecycle:
      preStop:
        exec:
          command: ["/bin/sh", "-c", "sleep 15"]
    ```
  </Accordion>

  <Accordion icon="check" title="Log to stdout/stderr">
    Make logs accessible via kubectl/Orphelix
    
    ```javascript
    // Good
    console.log("Message")
    
    // Bad (logs to file only)
    fs.writeFile("/var/log/app.log", "Message")
    ```
  </Accordion>
</AccordionGroup>
