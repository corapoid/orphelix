## Overview

Demo Mode lets you explore all Orphelix features using realistic mock data, without needing a Kubernetes cluster. Perfect for testing, presentations, training, or learning Kubernetes concepts.

<img src="/images/demo-mode.png" alt="Demo Mode" />

## What is Demo Mode?

Demo Mode provides:

  <Card title="Realistic Data" icon="database">
    Mock deployments, pods, nodes, and events that simulate a production environment
  </Card>
  <Card title="Full Features" icon="sparkles">
    All Orphelix features work in demo mode - explore without limitations
  </Card>
  <Card title="No Cluster Needed" icon="ban">
    No Kubernetes cluster or kubectl required
  </Card>

    Make changes safely - nothing affects real infrastructure

## Enabling Demo Mode

### Automatic Demo Mode

On first launch, Orphelix starts in Demo Mode automatically if no cluster is configured.

<Steps>
  <Step title="Start Orphelix">
    ```bash
    npm run orphelix start
    # or
    npm run dev
    ```
  </Step>

  <Step title="Open Browser">
    Navigate to [http://localhost:3000](http://localhost:3000)
    
    Or [http://orphelix.local:3000](http://orphelix.local:3000) if using CLI
  </Step>

  <Step title="Demo Mode Active">
    Header shows **"DEMO MODE"** badge
    
    All data is mocked
  </Step>
</Steps>

### Manual Switch to Demo Mode

If connected to a cluster, switch to demo mode:

<Steps>
  <Step title="Click Mode Badge">
    Click **"DEMO MODE"** or cluster name badge in header
  </Step>

  <Step title="Select Demo Mode">
    Toggle from "Real Cluster" to "Demo Mode"
  </Step>

  <Step title="Confirm">
    Page reloads with demo data
  </Step>
</Steps>

## Switching to Real Cluster

Ready to connect to a real cluster?

<Steps>
  <Step title="Click Demo Mode Badge">
    Click **"DEMO MODE"** badge in header
  </Step>

  <Step title="Select Real Cluster">
    Toggle to "Real Cluster"
  </Step>

  <Step title="Choose Context">
    Select kubectl context from dropdown
    
    (Requires kubeconfig configured)
  </Step>

  <Step title="Select Namespace">
    Choose namespace to monitor
    
    Default: "default"
  </Step>

  <Step title="Test Connection">
    Click **"Test Connection"** to verify cluster access
  </Step>

  <Step title="Switch Mode">
    Click **"Switch to Real Cluster"**
    
    Page reloads with live data
  </Step>
</Steps>

<Note>
  You can switch back to Demo Mode anytime by clicking the cluster name badge and selecting "Demo Mode"
</Note>

## Demo vs Real Cluster

Comparison:

| Feature | Demo Mode | Real Cluster |
|---------|-----------|--------------|
| **Data Source** | Mock data | Live Kubernetes API |
| **Actions** | Simulated | Real effects |
| **Requirements** | None | kubeconfig, cluster access |
| **Performance** | Fast (no API calls) | Depends on cluster |
| **Safety** | 100% safe | Actions affect cluster |
| **Learning** | Perfect | Requires caution |
| **Offline** | Works offline | Requires connectivity |
| **Auto-refresh** | Simulated | Real-time updates |

## Customizing Demo Data

Advanced: Modify demo data for custom scenarios:

<Steps>
  <Step title="Find Demo Data">
    Demo data in: `app/lib/demo/mock-data.ts`
  </Step>

  <Step title="Edit Mock Data">
    Modify deployments, pods, events, etc:
    
    ```typescript
    export const mockDeployments = [
      {
        name: "custom-app",
        namespace: "default",
        replicas: { desired: 3, available: 3 },
        status: "Available",
        // ... more fields
      },
      // Add more deployments
    ];
    ```
  </Step>

  <Step title="Rebuild">
    ```bash
    npm run build
    npm run orphelix restart
    ```
  </Step>

  <Step title="View Changes">
    Demo mode now shows your custom data
  </Step>
</Steps>

**Use cases for custom demo data:**
- Company-specific scenarios
- Training for specific architectures
- Testing with high resource counts
- Simulating specific failure modes

## Limitations

Demo Mode has some limitations:

<AccordionGroup>
  <Accordion icon="ban" title="Static Data">
    Demo data doesn't fully simulate cluster:
    - Changes don't persist between reloads
    - Resource creation not supported
    - No real-time events (simulated only)
  </Accordion>

  <Accordion icon="ban" title="No Real Actions">
    Actions are simulated:
    - Restart deployment doesn't actually restart
    - YAML changes don't apply to anything
    - PRs go to demo repository (or fail gracefully)
  </Accordion>

  <Accordion icon="ban" title="Limited Realism">
    Some behaviors simplified:
    - Metrics don't reflect real resource usage
    - Events don't follow real Kubernetes patterns exactly
    - No actual pod scheduling
  </Accordion>
</AccordionGroup>

<Note>
  These limitations are intentional - demo mode prioritizes safety and ease of use over perfect simulation
</Note>
